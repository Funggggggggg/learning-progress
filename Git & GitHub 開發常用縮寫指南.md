# Git & GitHub 開發常用縮寫指南

在 Git 的操作流程、指令以及 GitHub 的協作中，有許多常見的英文縮寫。簡單補充這些縮寫代表什麼，提升開發效率，也讓後續在閱讀文件或與他人溝通時可以更專業。

以下將這些縮寫分為三大類整理：

## 1. 指令與核心操作縮寫

這些縮寫是 Git 系統內建或開發環境中公認的標準表達方式。

| 縮寫 | 全稱 | 意義 / 用法 |
| :--- | :--- | :--- |
| **`m`** | Message | 用於 `git commit -m "..."`，代表提交訊息。 |
| **`amend`** | --amend | **修正最後一次提交**。用於補傳檔案或改錯字。實際上是建立新 Commit 覆蓋舊的，若已 Push 到雲端請謹慎使用。 |
| **`ff`** | Fast-forward | **快進式合併**。當分支可以順著路徑直接移動到目標位置時，Git 會自動使用的合併方式。 |
| **`HEAD`** | HEAD | **目前位置指標**。代表你現在正在哪一個分支或哪一次提交上。 |

---

## 2. GitHub 協作與工作流縮寫（業界最通用）

在 GitHub 討論區、Issue 或 Pull Request 中最常出現的開發者暗號。

| 縮寫 | 全稱 | 說明 |
| :--- | :--- | :--- |
| **`PR`** | Pull Request | **合併/拉取請求**。請求將你的程式碼合進主分支。GitHub 核心協作流程。 |
| **`LGTM`** | Looks Good To Me | **「我覺得可以」**。Reviewer（審核者）確認代碼沒問題，同意合併的標準回覆。 |
| **`WIP`** | Work In Progress | **開發中**。標記此 PR 尚未完成，提醒他人先不要合併，僅供進度同步。 |
| **`CI / CD`** | Continuous Integration / Deployment | **持續整合與部署**。指自動化測試、檢查程式碼並自動上線的整套流程。例：把程式碼推送到 GitHub 時，它會自動幫你檢查有沒有寫錯、跑測試，並直接把最新的網頁更新到網路上。 |
| **`feat / fix`** | Feature / Fix | **新功能 / 修復 Bug**。Commit 訊息最標準的前綴，用來分類這次更新的性質。 |

## 3. 專案開發與環境常用語

在管理專案套件或標示版本時最常看到的術語。

| 縮寫 | 全稱 | 說明 |
| :--- | :--- | :--- |
| **`semver`** | Semantic Versioning | **語意化版本**。如 `v1.2.3`（主版號.次版號.修訂號），是全球公認的版本命名規範。 |
| **`deps`** | Dependencies | **相依套件**。指專案執行所需要的外部工具包。例如 `Update deps` 代表更新套件。 |
| **`Changelog`** | Change Log | **更新日誌**。記錄每個版本具體改了什麼的文件，通常由 `feat` 和 `fix` 內容彙整而成。 |
