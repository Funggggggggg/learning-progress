# markdown與html比較

Markdown 的設計初衷是為了讓文件具備高度的可讀性，並能輕鬆轉換成網頁所使用的 HTML 格式。
可以想像成是「用簡單符號，讓純文字變得有結構、有格式」的寫作方式，也是工程師的日常語言之一。
且到目前為止 Markdown 還是運用的十分廣泛，如：

- GitHub（README、文件）
- Notion（筆記）
- Obsidian（知識管理）
- 各種部落格、文件系統

這篇作為新手，簡單整理markdown與html之間的比較關係。

## Markdown 與 HTML 語法對照表

以下將 Markdown 與 HTML 的語法對照與核心關係整理如下：

| 功能描述 | Markdown 語法 | 對應 HTML 標籤 | 預覽效果範例 |
| :--- | :--- | :--- | :--- |
| **第 1 級標題** | `# 標題內容` | `<h1>` | <h1>標題</h1> |
| **第 6 級標題** | `###### 標題內容` | `<h2>` | <h6>標題</h6> |
| **粗體** | `**文字**` | `<strong>` | **粗體文字** |
|  | `__文字__` | `<strong>` |  __粗體文字__ |
| **斜體** | `*文字*` | `<em>` | *斜體文字* |
| **超連結** | `[連結名稱](URL)` | `<a href="...">` | [連結名稱](https://...) |
| **圖片** | `![替代文字](圖片路徑)` | `<img src="..." alt="...">` | (圖片顯示) |
| **無序清單** | `- 項目` | `<ul><li>` | • 項目內容 |
| **有序清單** | `1. 項目` | `<ol><li>` | 1. 項目內容 |
| **程式碼區塊** | ` ```javascript ` | `<pre><code>` | 程式碼區塊 |

    <td>A</td>
    <td>B</td>
    <td>C</td>
</tr>

### 文件開頭的 Front Matter

Front Matter 是放在文件最開頭的一段 metadata（詮釋資料），用來描述這份文件本身的資訊，而不是文件的內容。它本身不是 Markdown 語法，而是各平台在 Markdown 之上加的擴充慣例。

#### 基本概念

就像一本書的版權頁，記錄書名、作者、出版日期等資訊，但讀者通常不會「讀」這頁，它是給系統處理用的。它能不能被解析、解析後做什麼，完全取決於你用的工具，且同一份檔案在不同平台上的行為不一定一致。它是為了讓工具處理文件而存在的，如果 .md 檔案有搭配特定平台或工具，它很有用；如果只是純粹寫筆記，加不加都無所謂。考慮到後續可能轉換平台，建議文章還是寫基礎 Front Matter 為未來保留彈性。

#### 實際好處

1. **文件管理更方便**：當你有大量 .md 檔案時，工具可以透過 Front Matter 的欄位做篩選、排序、搜尋，不需要打開每份文件才知道內容是什麼。

2. **自動化處理**
Jekyll、Hugo 這類靜態網站產生器會讀取 date、title、layout 等欄位，自動幫你排列文章順序、套用版型、產生目錄，不需要手動處理。

3. **平台功能整合**
在 HackMD 設定 tags 就能用標籤篩選筆記；在 Obsidian 設定屬性就能在知識圖譜中做連結，這些都依賴 Front Matter。

#### VS Code 和 GitHub 的解析系統一樣嗎？

不完全一樣：

|  | VS Code | GitHub |
| :--- | :--- | :--- |
|解析 Front Matter？|靠套件（如 markdownlint）|原生支援，直接隱藏|
|YAML 語法規範標準|YAML標準|YAML標準|
|解析後的用途|提示、lint 檢查|純隱藏，不做其他處理|

## 結語

用 VS Code 寫 markdown 語法很方便，可以下載相關套件後開分頁在旁做預覽。推送到 GitHub 儲存庫時，除了能練習 Git 指令、累積技術資產外，也可以直接在 GitHub 上面看筆記，且因兩者都是使用標準 YAML 語法規範較不會有不相通的問題。
